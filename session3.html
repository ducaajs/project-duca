03 Fix 1 — Context Window Crash
Root Cause — Sampler Panic on Truncated 43,289-Token Prompt

The pipeline was building prompts of approximately 43,289 tokens. With num_ctx 32768 set in the Duca Modelfile, Ollama silently truncated the prompt at the context ceiling and corrupted the context window. The sampler then had no valid tokens to work from and panicked with a fatal error on every single attempt.

Because the prompt size never changed between retries, the crash looped infinitely. The pipeline appeared to start normally — the cron fired, the agent loaded — and then died silently every time. The cron log reported status: ok, masking the failure entirely.

WARN: truncating input prompt limit=32768 prompt=43289 keep=4 new=32768
panic: failed to sample token
Fix — Confirmed via Three Independent Simulations, Five Consecutive Passing Tests Before Implementation

PARAMETER num_ctx 49152 in the Duca Modelfile. VRAM budget calculated and verified before the change was applied to confirm sufficient headroom on the RTX 3060 12GB. Stable across all subsequent pipeline runs with no truncation warnings and no sampler panics.

04 Fix 2 — Cross-Agent Messaging Blocked
Root Cause — Two Config Gates, Both Required, Neither Documented Together

sessions_send to a different agent was returning forbidden on every attempt. The failure had two separate root causes that both had to be resolved simultaneously. Neither fix alone was sufficient. This combined requirement does not appear anywhere in the official OpenClaw 2026.3.13 documentation.

Gate 1
Session Visibility
Default tools.sessions.visibility is "tree" — current session and spawned children only. Setting to "agent" is also insufficient. Cross-agent sends require "all".

Error without fix: Session send visibility is restricted.
Gate 2
Agent-to-Agent Allow List
tools.agentToAgent is disabled by default. The allow list must include both the source agent and all target agents — including "main". Omitting the orchestrator's own agentId blocks all sends silently.

Error without fix: Agent-to-agent messaging denied.
// Both blocks required simultaneously — neither works without the other
"tools": {
  "sessions": {
    "visibility": "all"
  },
  "agentToAgent": {
    "enabled": true,
    "allow": ["main", "job-coordinator", "job-researcher",
              "job-analyst", "job-writer", "job-auditor"]
  }
}

After any config change, existing sessions retain stale config snapshots even after a gateway restart. Required sequence: openclaw gateway restart followed by openclaw agent --agent main --message "/new"

05 Fix 3 — Broken Orchestration Pattern Replaced
Confirmed Bugs — sessions_spawn and sessions_yield Non-Functional in OpenClaw 2026.3.13

Issue #46298: sessions_yield immediately terminates isolated cron sessions instead of waiting for sub-agent completion.

Issue #49572: After sessions_spawn, the orchestrator is woken by the announce callback but aborts after exactly one LLM turn.

Both are 100% reproducible. The cron log reports status: ok — masking the failure entirely. The pipeline appeared healthy. Every run was dead on arrival.

Working Solution — sessions_send Blocking Wait Pattern, Five Consecutive Passing Runs Before Lock-In

Replace sessions_spawn + sessions_yield entirely. Each specialist agent runs as a persistent named session. The orchestrator sends a message directly to each session and waits for the reply. Confirmed working across five or more consecutive pipeline runs before the solution was locked into production.

// Add to your orchestrator's AGENTS.md

CRITICAL: Do NOT use sessions_spawn. Do NOT use sessions_yield.
Both are broken in OpenClaw 2026.3.13 and will kill the pipeline silently.

Use sessions_send with blocking wait instead:

1. sessions_send(sessionKey: "agent:worker-1:main", message: "[full briefing]", timeoutSeconds: 3600)
2. Wait for reply. Scrub output. Retry with same timeoutSeconds if needed.
3. Pass result forward to next agent the same way.
4. Each agent replies REPLY_SKIP to end the exchange cleanly.

If sessions_send returns status: timeout — agent is still running.
Call sessions_history on that session key to retrieve results.
