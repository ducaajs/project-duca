01 Project Overview

Project Duca is a fully local, autonomous job-search pipeline engineered for fit over volume — it surfaces a small set of high-fit roles each night rather than maximizing lead count, and no candidate data ever leaves the machine.

It runs unattended on a dedicated Ubuntu 24.04 workstation, drawing leads from three independent sources — major public job aggregators, forwarded email alerts, and directly from employers' own applicant-tracking systems — then verifies and enriches each lead, scores it against a fixed candidate profile with a local LLM, and builds per-lead application packets. Data and packets are filed to cloud storage; a nightly status alert reports what ran, when it finished, and whether the scoring model ran on the GPU or fell back to the CPU. Zero cloud inference. Zero paid API usage.

Duca is built on two principles: all data stays on local hardware, and a human stays in control of every decision. Cloud job tools and auto-applier bots expose sensitive career data to third-party servers and can trip applicant-tracking blacklists that silently kill applications. Duca keeps all processing on the local machine. The pipeline finds the signal and flags the risks; it never applies on anyone's behalf — every application is a human decision, and nothing is submitted without human review.

02 Governance & Control

Privacy and control are design requirements, not add-ons. The system runs locally, shares nothing without approval, and never acts on its own. Duca's role is to protect the user: surface the right openings, flag the dangers, and leave every decision to a human.

Privacy by Design
Data stays on the machine

All processing runs locally on dedicated hardware. Career history, targets, and documents never go to a third-party server, are never used to train a model, and carry no telemetry. The job searches themselves reach out — that's the work — but personal data does not.

Human in the Loop
It finds & flags — a human decides

Duca never applies for anything. The filters flag the dangers — ghost jobs, scams, salary games — and present vetted openings with the risks surfaced. The user controls where data goes and what gets shared, and tracks every application's outcome.

What Duca deliberately does NOT do
  • Auto-apply to anything — every application is a human decision, made by hand
  • Send career data to a third-party server, or use it to train a model
  • Run inference in the cloud — the model is 100% on-device
  • Emit telemetry or analytics, or depend on any paid API

04 Key Decisions & Tradeoffs
Optimized for Fit, Not Volume

Decision: Tune the whole pipeline for fewer, higher-fit leads in the target lane — never for daily lead count.

Why: A single strong 85% match is worth more than ten 40% long shots — the ten cost roughly 10× the research-and-apply hours with little chance of return. The pipeline is tuned for signal: a lane-targeted source adds ~36 on-target leads a night (off-lane roles excluded at the source), and metadata filters strip ~15–21 off-target leads each night before review. Tradeoff: lower headline volume — by design. Better work, not more work.

Deterministic Python + a Single LLM Stage

Decision: Confine the model to the one task only it can do — nuanced fit-scoring — and make every gathering, verification, and filtering stage plain, testable Python.

Why: An LLM is slow, nondeterministic, and priced per token; using it everywhere would make the pipeline unpredictable and costly. Keeping it to one stage means the run is reproducible, debuggable, and free to operate. Tradeoff: no free-form agent autonomy — a deliberate choice in favor of reliability.

Local Model Over Cloud APIs

Decision: Run all inference on a local GPU rather than a hosted LLM API.

Why: No rate limits or vendor lock-in, and — the point of the project — sensitive career data never leaves the machine. Tradeoff: capped by a single GPU's model size; mitigated with tight prompts and batching so the smaller model still scores accurately.

Cron Over a Long-Running Daemon

Decision: Schedule a nightly one-shot run with a file lock instead of a persistent service.

Why: Nothing to babysit — a scheduled process can't leak memory or wedge overnight, and a lock prevents overlapping runs and retry storms. Tradeoff: no real-time reaction, which a once-a-night job doesn't need.

Storage + Messaging Over a Custom Dashboard

Decision: Deliver results to cloud storage and a phone alert rather than building a web UI.

Why: Zero infrastructure to host or secure, the status reaches a phone where the decision gets made, and the data lands where the human review step already works. Tradeoff: less rich than a dashboard — but review happens off-box by design.

06 Critical Debugging
Silent System Failure — Root Cause Analysis

Problem: OpenClaw's api: "ollama" mode sends system prompts using role: "developer". Ollama silently drops unrecognized roles. Duca's 24,000-character system prompt vanished on every request with no error output.

Fix: Switched to api: "openai-completions" with an explicit compatibility block routing through role: "system". Full system prompt reception confirmed on first attempt after fix.

Community-Contributed Fixes — OpenClaw Issues #46298 & #49572

Problem: sessions_spawn and sessions_yield fail silently in OpenClaw 2026.3.13. No posted fix existed. Cross-agent messaging required two undocumented configuration gates with no mention in official documentation.

Fix: Built a working replacement using sessions_send with a blocking wait pattern. Published the first documented solutions to both open issues, along with the undiscovered configuration requirements.

Production Hardening — Scoring Cut from 72 to 8–15 Minutes

LLM scoring runtime reduced from 72 minutes to 8–15 minutes via direct API calls, adaptive batching, and split-retry with recursive halving. Context overflow (97KB merged JSON exceeding the 32K budget) resolved by restructuring pipeline stdout. Cross-company contamination in search enrichment fixed with strong-token relevance filtering.

Recall-Preserving Source Filter — Direct Sourcing

Problem: An early draft of the direct-sourcing filter discarded postings older than a set age as likely “ghost jobs.” External research on hiring patterns showed that senior and director-level roles routinely stay open for months — the exact roles the system is built to surface. The speed-oriented filter would have silently dropped its highest-value targets.

Fix: Replaced age-based dropping with a surface-not-drop model — nothing is auto-discarded on posting age; borderline leads are surfaced with reasons for review. Recall is favored over precision, with the review step refining the final set. Caught before deployment through research-first review, not a production incident.

07 Engineering Highlights
System Design & Architecture
  • A dedicated Ubuntu workstation (DucaBrain) built to run the pipeline and local model unattended
  • Multi-stage pipeline with deterministic Python stages and a single LLM scoring stage
  • Three independent lead sources — aggregator scraping, parsed email alerts, and direct-from-employer feeds — normalized into one lead shape and cross-deduplicated
  • Direct-from-employer sourcing via keyless applicant-tracking-system APIs, backed by a validated registry of 3,000+ company boards seeded from a public web-index crawl
  • 6 sender-specific email parsers across major job-alert providers, including multi-format handling for a single sender
  • Structured cloud-storage output with per-lead packet generation, shared automatically
Configuration & Integration
  • OAuth 2.0 headless authentication against a major cloud provider using a manual callback flow
  • Messaging integration with channel routing, allowlisting, and delivery acknowledgment
  • OpenClaw multi-agent configuration with model assignments, tool deny lists, and isolated workspaces
Open-Source Contributions
  • Posted first documented solutions to two open OpenClaw GitHub issues with no known public fix
  • Published a full technical writeup with root cause analysis and working replacement patterns
08 Limitations & Roadmap
  • Recall over precision, by design. Some noise reaches the human review step — a deliberate tradeoff to avoid discarding real openings, not a defect.
  • Single-GPU model ceiling. Scoring nuance is bounded by what fits on one video card; tight prompting and batching close most of the gap to a larger model.
  • Coverage tracks the registry. Direct-feed reach is only as broad as the validated board registry; long-tail discovery is a periodic step, not exhaustive.
  • Next: a held-out, hand-labeled set to measure scoring precision/recall; per-run analytics in the Morning Report; and broader regression coverage around the parsers.
